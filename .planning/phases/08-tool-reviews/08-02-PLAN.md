**content/tools_automation.py** (3 tools):
- `make`: Make (formerly Integromat). Focus on visual workflow builder, HTTP module flexibility, per-operation pricing model. Criticism: debugging complex scenarios is painful, error handling is opaque, community modules vary in quality. Comparison angle vs n8n (self-hosted, free) and Zapier (simpler but more expensive per task).
- `n8n`: Self-hosted workflow automation. 54% adoption among GTM Engineers. Focus on self-hosting advantage (no per-execution costs), code node flexibility, community nodes. Criticism: self-hosting requires DevOps knowledge, UI can lag on complex workflows, some nodes lack documentation. Note the massive adoption surge.
- `zapier`: No-code automation incumbent. Focus on ease of use, massive integration library (6,000+), multi-step Zaps. Criticism: per-task pricing gets expensive fast for GTM Engineers running high-volume workflows, limited data transformation capabilities, AI features feel bolted on. The "training wheels" verdict.

**content/tools_intent.py** (2 tools):
- `6sense`: Enterprise intent data platform. Focus on account identification, predictive analytics, ABM orchestration. Criticism: pricing is opaque and starts at $25K+/year, implementation takes months, data accuracy debates, SMB GTM Engineers can't justify the cost. Honest about the enterprise-only reality.
- `bombora`: B2B intent data via publisher co-op. Focus on Company Surge data, topic taxonomy, integration with CRMs/MAPs. Criticism: topic-level intent is noisy (a company "surging" on "CRM" could mean anything), requires significant filtering to be actionable, pricing unclear without sales call. Compare directly with 6sense.

**content/tools_analytics.py** (2 tools):
- `segment`: CDP/analytics. Focus on data collection, identity resolution, 400+ integrations, Protocols for data quality. Criticism: pricing escalates with MTUs (monthly tracked users), complexity overkill for most GTM Engineer use cases, Twilio acquisition created uncertainty about roadmap. Most GTM Engineers don't need a CDP.
- `posthog`: Open-source product analytics. Focus on self-hostable, session replay, feature flags, generous free tier (1M events/mo). Criticism: product analytics focus means it's not a replacement for marketing analytics, self-hosting requires infrastructure, some enterprise features are still maturing. The "developer-friendly" angle matters for GTM Engineers who code.

**content/tools_linkedin.py** (2 tools):
- `linkedin_sales_nav`: LinkedIn Sales Navigator. Focus on advanced search, lead/account lists, InMail credits, CRM sync. Criticism: expensive ($99+/mo), search filters miss many GTM Engineer prospects (role is too new), InMail response rates are declining, the "Social Selling Index" is vanity metrics. Most GTM Engineers use it but wish they didn't have to pay LinkedIn's toll.
- `phantombuster`: LinkedIn automation. Focus on profile scraping, connection requests, message sequences, data extraction. Criticism: LinkedIn actively fights automation (account restrictions), phantom credits get expensive for high-volume use, data extraction quality varies, ethical gray area. Be direct about the risks.
