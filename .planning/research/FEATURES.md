# Feature Landscape

**Domain:** Niche salary benchmark and career resource site (GTM Engineer vertical)
**Researched:** 2026-03-10
**Reference sites analyzed:** levels.fyi, Glassdoor, Built In, therevopsreport.com, salary.com

## Table Stakes

Features users expect. Missing = product feels incomplete or untrustworthy.

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| Salary index page with aggregate stats | Every salary site has a central hub showing median, average, range, and sample size. Users need the "headline number" immediately. | Low | Stats grid (median, avg, high, count) above the fold. therevopsreport.com pattern: 4 stat cards + definition block + seniority table. |
| Salary breakdown by seniority | Users always ask "what do I make at MY level?" Levels.fyi built their entire brand on leveling. | Low | 4 seniority tiers (Junior, Mid, Senior, Lead/Staff) with salary ranges. Each gets its own page with 1,200+ words of context. |
| Salary breakdown by location | Geographic variance is the second most common salary question. Built In has 40+ location pages. | Low | 15 city pages + Remote. Include cost-of-living context, not just raw numbers. |
| Salary comparison pages (vs other roles) | "GTM Engineer vs RevOps salary" is a high-intent search query. Glassdoor and levels.fyi both have role comparison features. | Medium | 10 comparison pages. Each needs side-by-side data tables, overlap analysis, career path context, and FAQ. |
| FAQ sections on content pages | Every salary and comparison page on levels.fyi and therevopsreport.com includes FAQ. Google uses these for featured snippets via FAQPage schema. | Low | 3-5 Q&A pairs per page. Must match visible content (FAQPage schema requirement). |
| Newsletter signup (multiple placements) | therevopsreport.com places signup CTAs in 4 locations: hero, inline mid-page, sticky scroll-trigger, footer. Industry standard for content sites. | Low | Resend + Cloudflare Worker. Contextual copy per placement ("Get weekly GTM Engineer salary data" not generic "Subscribe"). |
| Breadcrumb navigation | Standard UX pattern for hierarchical content. Also generates BreadcrumbList schema for rich results. | Low | Visual breadcrumbs + JSON-LD on every inner page. |
| Mobile-responsive design | 60%+ of salary searches happen on mobile. Non-negotiable. | Medium | 375px / 768px / 1024px breakpoints. Data tables must scroll horizontally or stack on mobile. |
| Sitemap.xml + robots.txt | Required for search engine crawling. Auto-generated by build system. | Low | Include all pages with accurate lastmod dates (not build time). Google ignores priority/changefreq, skip those. Max 50,000 URLs per file. |
| About page with real author | Salary data needs trust signals. Anonymous sites get discounted by both users and Google (E-E-A-T). | Low | Rome Thorndike bio, credentials, mission statement. Person schema markup. |
| Internal linking (3+ per page) | Hub-spoke linking is critical for programmatic SEO. Every salary page must link to related breakdowns, comparisons, and the index. | Low | Automated via build system. "Related salary data" section at bottom of every page. Category hubs link to all children; children link back to hub and to siblings. |
| Methodology / data sources page | Users and Google both need to know where numbers come from. Levels.fyi, Glassdoor, and salary.com all have methodology pages. | Low | Explain data collection, sample sizes, update frequency, limitations. Builds credibility. |
| Clean URL structure | `/salary/by-location/san-francisco/` not `/pages/salary-sf.html`. Every salary site uses semantic, directory-based URLs. | Low | Python build generates directory-based URLs with `index.html` files. |
| Fast page loads (<2s) | Static HTML should be near-instant. Career/salary users bounce fast. | Low | No JS framework overhead. Google Fonts are the heaviest asset. Target Lighthouse 90+. |
| Privacy policy + Terms | Legal necessity for email collection. Google expects it for E-E-A-T. | Low | Static pages. Required before newsletter can launch. |

## Differentiators

Features that set the product apart. Not expected by default, but create competitive advantage.

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| Salary by company stage | No competitor breaks down GTM Engineer pay by Seed / Series A / Series B / Growth / Enterprise. Comp varies wildly by stage, and this is uniquely relevant to a startup-heavy role. | Low | 5 pages. Levels.fyi only does it by company name, not stage. Built In doesn't do it at all. Pure whitespace. |
| Email-gated salary calculator | Built In uses "share your salary to unlock data." Levels.fyi gates negotiation coaching behind $1,250+ packages. A simple calculator that shows free median + range but gates the full percentile breakdown (by location + seniority + stage combo) drives email capture without hurting SEO. | Medium | The "unlock" moment is the lead magnet. JS-only calculator, no backend needed. Free preview stats keep the page indexable. |
| Contextual newsletter CTAs | Most sites use the same generic CTA everywhere. Contextual CTAs convert 2-3x better. On a salary-by-location page: "Get weekly salary shifts for [city]." On a comparison page: "See how GTM Engineer comp stacks up this week." | Low | Template function takes `context` param. 4-6 CTA variants mapped to page types. |
| Definition blocks on every page | therevopsreport.com opens each salary page with a styled definition block explaining the topic. Serves double duty: answers the user's implicit "what is this?" question and provides unique opening content that separates the page from boilerplate. | Low | `definition_block(term, definition)` template function. Each page gets a unique definition tailored to its specific topic. |
| Deep content (1,200-2,000 words per page) | Programmatic SEO in 2026 requires 500+ unique words minimum per page, but 1,200+ is where pages actually rank. Most salary sites phone it in with 200-word template swaps around a data table. | Medium | Market context, compensation drivers, career trajectory, negotiation tips, FAQ. Each page must have 30-40% unique content beyond the shared template structure. |
| Week-over-week salary trend data | Levels.fyi shows YoY trends in annual reports only. No niche salary site shows weekly shifts. The scraper pipeline already computes WoW diffs for the newsletter. Surface that delta data on salary pages. | Medium | Requires `previous_market_snapshot.json` data. Show trend arrows and delta percentages. "GTM Engineer salaries in SF are up 3.2% this month." Builds a reason to return. |
| Job posting growth stats | 205% YoY growth is the anchor stat that makes GTM Engineer compelling. No competitor tracks this. Positions the site as the definitive authority on the role's trajectory. | Low | Scraper data feeds total posting counts, new listings/month, top hiring companies. Display as stats grid on homepage and salary index. |
| Vendor-neutral tool reviews (Wave 2) | gtmehq.com is a Clay template shop. Every other GTM tool resource is vendor-affiliated. Honest, independent reviews with actual criticism are rare. | High | 30 tool reviews across 8 categories. Affiliate links exist but don't bias scores. Differentiation through honesty. |
| 50-term glossary (Wave 3) | "What is a GTM Engineer" has weak search results. Definitional content captures top-of-funnel traffic and builds topical authority. | Medium | Programmatic pages from structured data. Cross-linked to salary/tool pages for internal link equity. |
| Job board with live scraper data (Wave 4) | Real-time GTM Engineer job postings from the unified scraper. Competitors have nothing. | High | Scraper exports JSON, build system generates static job cards. Auto-updated on cron. |
| Tool stack correlation with salary | "GTM Engineers who list Clay + Python earn 18% more." No competitor does this. Requires enriched job posting data mapping tool requirements to salary ranges. | High | Wave 4+ feature. Needs sufficient posting volume with both salary and tool data. Killer differentiator once data exists. |

## Anti-Features

Features to explicitly NOT build. These seem obvious but would hurt the site.

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| User-submitted salary data / "contribute your salary" | Levels.fyi and Glassdoor rely on user submissions because they cover thousands of roles across millions of companies. GTME Pulse covers one role. User submissions introduce noise, require moderation, and the site doesn't have the traffic to generate meaningful volume yet. | Use job posting data from the scraper pipeline. It's verifiable, consistent, and updates automatically. Consider user submissions only if/when the site exceeds 10K monthly visitors. |
| Full content gating / paywall | Gating salary pages behind login kills SEO. Google can't index gated content. Glassdoor's "sign up to see more" pattern works because they have massive brand recognition and domain authority. A new site cannot afford this tradeoff. | Keep all content free and indexable. Gate only the interactive calculator's full results (percentile breakdowns). Capture emails through value-add (newsletter, calculator unlock) not access restriction. |
| Estimated salary schema markup | Google deprecated Occupation/estimatedSalary structured data in September 2025. It is no longer processed or displayed in search results. Implementing it is wasted effort. | Use BreadcrumbList, FAQPage, Organization, WebSite, and Article schemas instead. These are actively supported and generate rich results. |
| Thin programmatic pages | Salary pages that just swap a number into a template. "The average GTM Engineer salary in Austin is $145K." That's a Google snippet answer, not a page worth visiting. Pages under 300 words with only variable swaps risk spam filtering in 2026. | Every page needs market context, compensation drivers (why Austin pays X vs Denver), career trajectory advice, negotiation leverage points, and FAQ. 1,200+ words minimum with 30-40% unique content. |
| Dark patterns for email capture | Exit-intent popups, countdown timers, "only 3 spots left" urgency. GTM Engineers build automation systems for a living. They will see through manufactured urgency immediately and lose trust. | Clean inline CTAs with honest value props. "Weekly salary data and market shifts. No spam." The audience respects directness. |
| Real-time salary search / database query UI | Levels.fyi and Glassdoor have millions of data points that justify a search/filter interface. GTME Pulse has curated data for one role across 45 pages. A search box returning 45 results looks empty and undermines perceived data depth. | Pre-built pages organized by seniority, location, stage, and comparison. Clear navigation beats empty search. Add search only when page count exceeds 200+. |
| User accounts / login system | Massive complexity (auth, sessions, password reset) for zero Wave 1 value. Email capture via Resend is sufficient for lead generation. | Resend Audiences for subscriber management. No user profiles needed. |
| Comment system | Attracts spam, requires moderation infrastructure, adds JS weight. Career/salary reference sites don't need discussion. | Contact email on About page. Community building happens via newsletter, not comments. |
| Display ads | Banner ads on a 200-page niche site generate pennies ($1-5/day) and destroy credibility with a high-income B2B audience ($130K-$250K earners). | Affiliate links in tool reviews (Wave 2), newsletter sponsorships, job board paid listings. Revenue comes from the audience's purchasing power, not ad impressions. |
| Client-side search | Unnecessary until 500+ pages. Internal linking and clear nav are better discovery. Adds JS weight and index maintenance. | Well-structured nav + related links sections. |
| Cookie consent banner | No cookies set by the site itself. Google Fonts loaded via link tags (no cookies). | Don't set cookies = no banner needed. |
| Animated transitions / scroll effects | Adds JS weight, slows perceived performance, distracts from data. This is a reference site, not a product marketing page. | Clean, fast page loads. Minimal CSS transitions on hover states only. |
| Social share buttons | Cluttered, rarely clicked (<0.5% click rate on most sites), add third-party tracking scripts. | Clean design. People copy URLs. |
| Dark mode toggle UI | tokens.css already has dark mode via `prefers-color-scheme` media query. A manual toggle adds JS complexity for minimal value on a data reference site. | Let OS preference handle it automatically. |

## Feature Dependencies

```
Build system (3-file Python split) -> ALL pages
  |-- templates.py (HTML components, schema helpers) -> Every page type
  |-- nav_config.py (navigation, footer, constants) -> Every page type
  +-- CSS architecture (tokens + components + styles) -> Every page type

Newsletter infrastructure (Cloudflare Worker + Resend) -> Newsletter CTAs on all pages
  +-- Email-gated calculator (requires signup flow working first)

Salary index page -> All salary sub-pages (hub must exist before spokes)
  |-- By-seniority pages (4)
  |-- By-location pages (15)
  |-- By-company-stage pages (5)
  +-- Vs-comparison pages (10)

Salary data (hardcoded in build.py) -> All salary pages
  +-- JSON data files (data/) -> salary data migration (future)

Scraper pipeline (existing, external) -> Job posting stats, WoW trends
  +-- Newsletter automation (Wave 4, depends on scraper exports)

Homepage -> References all content verticals
  +-- Should be built LAST in Wave 1 (needs salary pages to exist for linking)
```

## Salary Page Section Structure (Recommended)

Based on analysis of levels.fyi, therevopsreport.com, Built In, and Glassdoor.

### Salary Index Page
1. Breadcrumb (Home > Salaries)
2. H1 title + subtitle
3. Stats grid: median salary, average salary, highest reported, sample size (jobs analyzed)
4. Definition block: what GTM Engineer compensation includes (base, equity, bonus)
5. Seniority breakdown table: level name, avg salary, range, job count per tier
6. Navigation cards to sub-sections (by seniority, by location, by stage, calculator)
7. Methodology summary paragraph
8. FAQ section (3-5 Q&A)
9. Newsletter CTA (contextual: "Get weekly salary shift data")
10. Related links (comparisons, career guides when available)

### Salary Breakdown Pages (seniority, location, stage)
1. Breadcrumb (Home > Salaries > By [Dimension] > [Specific])
2. H1 title + subtitle
3. Stats grid: median, range (10th-90th percentile), sample size
4. Definition block (contextual to the specific breakdown)
5. Detailed analysis (1,200+ words): market context, comp drivers, negotiation tips specific to this dimension
6. Comparison table: this breakdown vs others in same category (e.g., SF vs NYC vs Austin)
7. Related breakdowns: links to other entries in same category
8. FAQ section (3-5 Q&A)
9. Newsletter CTA (contextual)
10. Related links: comparisons, calculator, index

### Salary Comparison Pages (GTM Engineer vs X)
1. Breadcrumb (Home > Salaries > Comparisons > GTM Engineer vs [Role])
2. H1 title ("GTM Engineer vs RevOps Salary")
3. Side-by-side stats table: both roles with median, range, growth rate
4. Definition blocks: both roles defined clearly
5. Detailed comparison (1,500+ words): responsibilities, skills overlap, career paths, day-to-day differences
6. Salary range overlap analysis (text-based with data, no JS charts needed)
7. "Which role fits you?" decision framework with concrete criteria
8. FAQ section (5+ Q&A, high featured snippet potential)
9. Newsletter CTA (contextual)
10. Related comparisons + salary breakdowns

### Salary Calculator Page
1. Breadcrumb
2. H1 title + value prop subtitle
3. Free preview: aggregate stats visible without email (keeps page indexable)
4. Calculator inputs: seniority level, location, company stage (dropdowns, no JS framework)
5. Free result: median salary for selection
6. Email gate: "Enter email for full percentile breakdown (10th-90th) + comp drivers"
7. Full results (post-email): percentile distribution, similar role comparison, negotiation data points
8. Methodology link
9. FAQ section
10. Related salary pages

## Newsletter Signup Patterns (Recommended)

Based on therevopsreport.com analysis and conversion optimization research:

| Placement | Trigger | Copy Pattern | Priority |
|-----------|---------|-------------|----------|
| Hero section (homepage) | Immediate, above fold | "Weekly GTM Engineer salary data, market shifts, and tool intel." | Must have |
| Inline mid-page (salary pages) | After seeing data, natural reading pause | "Get salary shift alerts for [city/level]." Context-specific. | Must have |
| Dedicated /newsletter page | Direct navigation or linked from CTAs | Full value prop: what you get, frequency, sample issue preview | Must have |
| Footer | Always visible, secondary | Short form with one-line hook | Must have |
| Calculator results | After email gate conversion | "Results sent to your inbox + weekly updates" (dual-purpose capture) | Must have (tied to calculator) |

## Programmatic Page Generation Patterns

Based on research into how salary sites scale content in 2026:

### What works
- **Template + unique prose hybrid:** Structural layout is templated (stats grid, table, FAQ container, related links). But 30-40% of each page must be genuinely unique written content. This is the line between ranking pages and spam-filtered ones.
- **Hub-spoke architecture:** Index page links to all children. Children link back to hub AND to 2-3 siblings. This distributes link equity and prevents orphan pages. Every page reachable within 2 clicks from homepage.
- **Data-driven but human-written:** Stats come from structured data (dicts/JSON). Context paragraphs are written per-page in content modules or inline in build.py. The build system merges both at generation time.
- **Content depth thresholds:** Under 300 words = spam risk. 500 words = minimum to exist. 1,200+ words with genuine analysis = where pages consistently rank. therevopsreport.com targets 1,200-2,000 per salary page.
- **Unique meta descriptions:** Every programmatic page needs a unique meta description. Duplicate descriptions across 15 city pages will get de-indexed. Template the structure but vary the content.

### What doesn't work
- **Pure template swap:** "The average [ROLE] salary in [CITY] is [NUMBER]" repeated 50 times with only variables changed. Google filters these aggressively. Every page must have unique analysis paragraphs.
- **Stale data:** Salary data outdated by 6+ months erodes trust and earns negative engagement signals. Plan for quarterly data refreshes minimum, monthly if scraper supports it.
- **Orphan pages:** Programmatic pages not linked from any hub page get zero crawl priority. The sitemap alone is not enough. Structural links matter.
- **Identical FAQ across pages:** Copy-pasting the same 5 Q&A on every page defeats the purpose. Each page should have 2-3 shared structural questions and 2-3 unique to that specific topic.

## MVP Recommendation (Wave 1)

Prioritize in this order:
1. **Build system skeleton** (3-file split, CSS architecture) - foundation for everything
2. **Newsletter infrastructure** (Cloudflare Worker + Resend) - must work before calculator can gate
3. **Salary index page** (hub page, stats grid, seniority table, nav cards) - the anchor hub
4. **4 salary-by-seniority pages** - highest search volume after index
5. **15 salary-by-location pages** - most programmatic leverage (template + unique city context)
6. **5 salary-by-company-stage pages** - unique differentiator no competitor has
7. **10 salary-vs-comparison pages** - highest-intent long-tail queries
8. **Salary calculator page** (email-gated full results) - email capture mechanism
9. **Methodology page** - trust and credibility signal
10. **Homepage, About, Privacy, Terms, 404** - standard pages (homepage last, needs content to link to)

Defer to later waves:
- **Tool reviews/comparisons (Wave 2):** Separate content vertical, significant writing effort per page
- **Career guides (Wave 3):** Important but salary data is the anchor differentiator
- **Glossary (Wave 3):** Top-of-funnel capture, not core value prop
- **Job board (Wave 4):** Requires scraper pipeline maturity (only 7 tagged postings currently)
- **WoW trend data on salary pages:** Needs scraper running consistently for meaningful week-over-week diffs
- **Weekly email automation (Wave 4):** Need subscriber base first
- **Sticky scroll CTA:** Nice-to-have; inline CTAs cover the conversion path for Wave 1
